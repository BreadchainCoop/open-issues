safety-net #14

Change Contestation Threshold Logic

Current Behavior

  • Any contest triggers voting phase
  • Requires all members to vote
  • 66% yes votes needed for approval
  • Can stall on low participation

Proposed Change

  • Auto-approve unless >33% contest
  • No voting phase needed
  • Simpler, faster withdrawals
  • Minority can still block bad actors

Why This Issue

A governance mechanism redesign for on-chain voting systems. The issue includes before/after Mermaid sequence diagrams showing exact flow changes. You'll implement threshold-based approval logic, handle edge cases like abstentions and member churn, and consider Sybil attack resistance. Practical experience with DAO governance patterns.

Governance Solidity Voting Systems Game Theory Security
